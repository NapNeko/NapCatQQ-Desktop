# NapCat WebUI 接口调试机制总结

## 1. 总览
NapCat 的 WebUI「接口调试」功能，整体不是“前端写死接口列表 + 直接发 HTTP 请求”这么简单，而是一套 **后端元数据导出 + 调试适配器转发 + Action 真实执行** 的机制。

可以把它理解成三层：

1. **接口列表从后端动态获取**
2. **请求示例优先取 Action 自带示例，兜底再按 schema 自动生成**
3. **真正调用时，不是浏览器直接打外部 OneBot 服务，而是经由 WebUI 后端的调试适配器，在当前进程内执行 OneBot Action**

---

## 2. 接口列表是怎么来的

前端会调用 [`fetchOneBotHttpApi()`](packages/napcat-webui-frontend/src/const/ob_api/index.ts:15)，请求 [`/api/Debug/schemas`](packages/napcat-webui-frontend/src/const/ob_api/index.ts:17)。

后端在 [`router.get('/schemas', ...)`](packages/napcat-webui-backend/src/api/Debug.ts:26) 中处理这个请求。它会：

1. 获取当前 OneBot 上下文 [`WebUiDataRuntime.getOneBotContext()`](packages/napcat-webui-backend/src/api/Debug.ts:28)
2. 遍历 [`ActionName`](packages/napcat-webui-backend/src/api/Debug.ts:35)
3. 从 [`obContext.actions.get(actionName)`](packages/napcat-webui-backend/src/api/Debug.ts:39) 中拿到实际注册的 Action
4. 导出这些 Action 的元数据：
   - [`actionSummary`](packages/napcat-webui-backend/src/api/Debug.ts:43)
   - [`actionDescription`](packages/napcat-webui-backend/src/api/Debug.ts:43)
   - [`payloadSchema`](packages/napcat-webui-backend/src/api/Debug.ts:44)
   - [`returnSchema`](packages/napcat-webui-backend/src/api/Debug.ts:45)
   - [`payloadExample`](packages/napcat-webui-backend/src/api/Debug.ts:46)
   - [`actionTags`](packages/napcat-webui-backend/src/api/Debug.ts:47)

### 结论
- **不是前端写死 API 列表**
- **是后端把当前运行时已注册的 Action 动态导出来给前端**

---

## 3. 这些接口定义本身是不是写死的

前端页面不是写死的，但每个接口的“定义信息”确实是在后端代码里声明的。

所有接口都基于 [`OneBotAction`](packages/napcat-onebot/action/OneBotAction.ts:47) 这个基类。它定义了：

- [`payloadSchema`](packages/napcat-onebot/action/OneBotAction.ts:51)
- [`returnSchema`](packages/napcat-onebot/action/OneBotAction.ts:52)
- [`payloadExample`](packages/napcat-onebot/action/OneBotAction.ts:53)
- [`returnExample`](packages/napcat-onebot/action/OneBotAction.ts:54)
- [`actionSummary`](packages/napcat-onebot/action/OneBotAction.ts:55)
- [`actionDescription`](packages/napcat-onebot/action/OneBotAction.ts:56)
- [`actionTags`](packages/napcat-onebot/action/OneBotAction.ts:57)

具体接口会在自己的 Action 类中覆写这些字段，例如：
- [`GetFriendList`](packages/napcat-onebot/action/user/GetFriendList.ts:18)
- [`SendMsg`](packages/napcat-onebot/action/msg/SendMsg.ts:503)

### 结论
- **前端展示是动态的**
- **接口描述、schema、示例来自后端 Action 类的声明式定义**

## 4. 选中某个 API 后，测试 JSON 怎么来的

生成请求 JSON 时有两级优先级。

### 第一优先级：直接使用 [`payloadExample`](packages/napcat-onebot/action/OneBotAction.ts:53)
在调试页的 [`executeCommand()`](packages/napcat-webui-frontend/src/pages/dashboard/debug/http/index.tsx:111) 中，前端会优先判断当前接口是否带有 [`payloadExample`](packages/napcat-webui-frontend/src/pages/dashboard/debug/http/index.tsx:115)。

如果有，就直接：
- [`JSON.stringify(item.payloadExample, null, 2)`](packages/napcat-webui-frontend/src/pages/dashboard/debug/http/index.tsx:116)

在具体调试组件的 [`useEffect()`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:179) 里也是一样：
- 如果有 [`data.payloadExample`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:180)
- 就设置到请求编辑器 [`setRequestBody(JSON.stringify(data.payloadExample, null, 2))`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:181)

### 第二优先级：按 schema 自动生成默认 JSON
如果没有示例，则调用 [`generateDefaultFromTypeBox()`](packages/napcat-webui-frontend/src/utils/typebox.ts:150)。

主要规则如下：
- 有 [`const`](packages/napcat-webui-frontend/src/utils/typebox.ts:176) 用常量
- 有 [`default`](packages/napcat-webui-frontend/src/utils/typebox.ts:177) 用默认值
- 有 [`enum`](packages/napcat-webui-frontend/src/utils/typebox.ts:178) 取第一个
- 联合类型优先挑简单值 [`anyOf` / `oneOf`](packages/napcat-webui-frontend/src/utils/typebox.ts:181)
- 对象递归生成字段 [`type === 'object'`](packages/napcat-webui-frontend/src/utils/typebox.ts:200)
- 数组返回空数组 [`type === 'array'`](packages/napcat-webui-frontend/src/utils/typebox.ts:215)
- 字符串返回空串 [`type === 'string'`](packages/napcat-webui-frontend/src/utils/typebox.ts:220)
- 数字返回 0 [`type === 'number' || type === 'integer'`](packages/napcat-webui-frontend/src/utils/typebox.ts:221)
- 布尔返回 false [`type === 'boolean'`](packages/napcat-webui-frontend/src/utils/typebox.ts:222)

### 一个细节
在调试面板里点击“生成示例”按钮时，走的是 [`generateDefaultFromTypeBox()`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:347)，也就是“按 schema 重生成占位 JSON”，不一定会优先保留原始 [`payloadExample`](packages/napcat-onebot/action/OneBotAction.ts:53)。

### 结论
请求 JSON 的来源是：
- **优先使用 Action 自带的 [`payloadExample`](packages/napcat-onebot/action/OneBotAction.ts:53)**
- **没有示例时再根据 [`payloadSchema`](packages/napcat-onebot/action/OneBotAction.ts:51) 自动生成**

## 5. HTTP 调试是怎么调用的

### 5.1 页面初始化：先创建调试适配器
HTTP 调试页加载时，在 [`useEffect()`](packages/napcat-webui-frontend/src/pages/dashboard/debug/http/index.tsx:45) 里会同时做两件事：

1. 获取接口 schema [`fetchOneBotHttpApi()`](packages/napcat-webui-frontend/src/pages/dashboard/debug/http/index.tsx:51)
2. 请求 [`/api/Debug/create`](packages/napcat-webui-frontend/src/pages/dashboard/debug/http/index.tsx:52) 创建调试会话

后端对应逻辑在 [`router.post('/create', ...)`](packages/napcat-webui-backend/src/api/Debug.ts:316)。
它会调用 [`debugAdapterManager.getOrCreateAdapter()`](packages/napcat-webui-backend/src/api/Debug.ts:318)，返回：
- [`adapterName`](packages/napcat-webui-backend/src/api/Debug.ts:320)
- [`token`](packages/napcat-webui-backend/src/api/Debug.ts:321)

这里创建出来的是一个专用的 [`DebugAdapter`](packages/napcat-webui-backend/src/api/Debug.ts:62)。

### 5.2 点击发送：前端把 action 和 params 发给后端
在 [`sendRequest()`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:89) 中，如果当前存在 [`adapterName`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:98)，前端会请求：

- [`/api/Debug/call/${adapterName}`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:100)

并携带：
- [`action: path`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:101)
- [`params: parsedRequestBody`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:102)

也就是说，前端发给后端的是：
- 要调用哪个 OneBot API
- 参数是什么

而不是一条真正意义上的“外部 HTTP OneBot 请求”。

### 5.3 后端接收：转交给调试适配器
后端在 [`handleCallApi()`](packages/napcat-webui-backend/src/api/Debug.ts:333) 中处理这个请求：

1. 取出 [`adapterName`](packages/napcat-webui-backend/src/api/Debug.ts:335)
2. 找到对应 adapter [`debugAdapterManager.getAdapter(adapterName)`](packages/napcat-webui-backend/src/api/Debug.ts:337)
3. 读取 [`action`](packages/napcat-webui-backend/src/api/Debug.ts:348) 和 [`params`](packages/napcat-webui-backend/src/api/Debug.ts:348)
4. 调用 [`adapter.callApi()`](packages/napcat-webui-backend/src/api/Debug.ts:349)

### 5.4 调试适配器执行真实 Action
[`DebugAdapter.callApi()`](packages/napcat-webui-backend/src/api/Debug.ts:148) 的逻辑是：

1. 从 [`this.actions.get(actionName)`](packages/napcat-webui-backend/src/api/Debug.ts:151) 找到已注册的 Action
2. 调用这个 Action 的 [`handle()`](packages/napcat-webui-backend/src/api/Debug.ts:157)

而 [`handle()`](packages/napcat-onebot/action/OneBotAction.ts:90) 会：
- 先执行参数校验 [`check()`](packages/napcat-onebot/action/OneBotAction.ts:67)
- 再执行实际业务 [`_handle()`](packages/napcat-onebot/action/OneBotAction.ts:118)
- 最后包装成标准返回 [`OB11Response.ok()`](packages/napcat-onebot/action/OneBotAction.ts:36) 或 [`OB11Response.error()`](packages/napcat-onebot/action/OneBotAction.ts:40)

### 结论
HTTP 调试的主链路是：

[`OneBotApiDebug.sendRequest()`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:89)
→ [`/api/Debug/call/:adapterName`](packages/napcat-webui-backend/src/api/Debug.ts:357)
→ [`handleCallApi()`](packages/napcat-webui-backend/src/api/Debug.ts:333)
→ [`DebugAdapter.callApi()`](packages/napcat-webui-backend/src/api/Debug.ts:148)
→ [`OneBotAction.handle()`](packages/napcat-onebot/action/OneBotAction.ts:90)
→ [`OneBotAction._handle()`](packages/napcat-onebot/action/OneBotAction.ts:118)

所以它本质是：
**前端请求 WebUI 后端，后端在当前 NapCat 进程内部真实执行对应 Action。**

## 6. 旧逻辑回退：也支持直接请求外部地址
在 [`sendRequest()`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:129) 里还有一个旧分支。

如果没有 [`adapterName`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:98)，会：
1. 根据 [`httpConfig.url`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:130) 拼 URL
2. 把当前 [`path`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:131) 挂到 pathname
3. 直接 [`request.post()`](packages/napcat-webui-frontend/src/components/onebot/api/debug.tsx:133)

这说明历史上也支持直接打外部 HTTP 服务。

但从当前初始化逻辑看，页面会优先走 [`/api/Debug/create`](packages/napcat-webui-frontend/src/pages/dashboard/debug/http/index.tsx:52)，因此现在主流程一般还是使用 [`DebugAdapter`](packages/napcat-webui-backend/src/api/Debug.ts:62)。

## 7. WebSocket 实时调试怎么调用

实时调试与 HTTP 调试类似，也是通过调试 adapter 完成，只是通信方式改成了 WebSocket。

### 7.1 前端建立 WS 连接
前端在 [`useWebSocketDebug()`](packages/napcat-webui-frontend/src/hooks/use-websocket-debug.ts:13) 中调用 [`useWebSocket()`](packages/napcat-webui-frontend/src/hooks/use-websocket-debug.ts:24)，并通过 [`queryParams.access_token`](packages/napcat-webui-frontend/src/hooks/use-websocket-debug.ts:40) 带 token。

### 7.2 后端升级连接
后端入口是 [`handleDebugWebSocket()`](packages/napcat-webui-backend/src/api/Debug.ts:395)。
它会：
1. 读取 [`adapterName`](packages/napcat-webui-backend/src/api/Debug.ts:397)
2. 读取 [`token`](packages/napcat-webui-backend/src/api/Debug.ts:398)
3. 校验 token [`adapter.validateToken(token)`](packages/napcat-webui-backend/src/api/Debug.ts:434)
4. 调用 [`adapter.addWsClient()`](packages/napcat-webui-backend/src/api/Debug.ts:445)

### 7.3 双向调试逻辑
- **服务端事件推送**：调用 [`DebugAdapter.onEvent()`](packages/napcat-webui-backend/src/api/Debug.ts:133)，把事件广播给 WebSocket 客户端
- **前端发消息调用 API**：进入 [`handleMessage()`](packages/napcat-webui-backend/src/api/Debug.ts:160)，解析收到的 JSON 后，执行对应 Action 的 [`websocketHandle()`](packages/napcat-onebot/action/OneBotAction.ts:104)

### 结论
WebSocket 实时调试同样不是 mock，而是：
- 通过调试 adapter 接入当前运行中的 OneBot 环境
- 真实接收事件
- 真实执行 Action

## 8. 整体结构图

```mermaid
flowchart TD
  A[WebUI 接口调试页] --> B[/api/Debug/schemas]
  B --> C[后端遍历 actions 注册表]
  C --> D[返回 schema / example / tags]

  A --> E[/api/Debug/create]
  E --> F[创建或复用 DebugAdapter]

  A --> G[/api/Debug/call/:adapterName]
  G --> F
  F --> H[this.actions.get(actionName)]
  H --> I[OneBotAction.handle()]
  I --> J[参数校验 check()]
  I --> K[业务执行 _handle()]
  K --> L[返回调试结果]

  M[实时调试 WS] --> N[handleDebugWebSocket()]
  N --> F
  F --> O[onEvent 推送事件]
  F --> P[handleMessage 处理前端发来的 action]
  P --> Q[OneBotAction.websocketHandle()]
```

## 9. 最终结论

NapCat WebUI 的接口调试机制可以总结为：

1. **接口列表不是前端写死，而是后端动态导出当前已注册的 Action 元数据**
2. **请求 JSON 优先来自 [`payloadExample`](packages/napcat-onebot/action/OneBotAction.ts:53)，没有示例时再按 [`payloadSchema`](packages/napcat-onebot/action/OneBotAction.ts:51) 自动生成**
3. **调试调用不是浏览器直接去打外部 OneBot 服务，而是经由 WebUI 后端的 [`DebugAdapter`](packages/napcat-webui-backend/src/api/Debug.ts:62) 在当前进程里真实执行 [`OneBotAction`](packages/napcat-onebot/action/OneBotAction.ts:47)**
4. **HTTP 调试和 WebSocket 实时调试，本质上都走同一套 Action 执行体系，只是传输通道不同**

因此，这套机制的本质可以概括为：

**“Action 元数据驱动的调试 UI + 基于 DebugAdapter 的进程内真实调用代理”。**
