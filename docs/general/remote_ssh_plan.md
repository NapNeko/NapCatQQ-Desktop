# NapCatQQ Desktop 远程 SSH 直连方案规划

## 1. 目标

本规划用于明确 NapCatQQ Desktop 的“本地 Windows Desktop + 远程 Linux 执行端”最优路线。

当前选定路线为：

- 本地保留现有桌面运行能力
- 远程侧**不引入常驻 agent / 守护进程 / 独立服务端**作为第一阶段前提
- 由 Desktop 通过 SSH/SFTP 直接管理远程 Linux 主机
- 用户理想使用路径保持为：**填写 SSH 信息 → 测试连接 → 一键部署 → 一键管理**

## 2. 为什么选择 SSH 直连

相比“先在远端安装一个独立服务再交互”，SSH 直连更适合当前阶段：

### 2.1 优势

- 用户理解成本最低，只需要已有 SSH 访问能力
- 不需要额外开放远程端口
- 不需要维护第二个长期驻留的软件产品
- 远程安全边界更容易解释和收敛
- 更适合当前项目的迭代速度和实现成本

### 2.2 当前不选择远端常驻服务的原因

- 首次部署仍然离不开 SSH 或脚本引导
- 会引入新的认证、鉴权、协议版本、升级回滚、安全暴露面问题
- 对当前项目来说，复杂度增长过快，超出第一阶段收益

## 3. 产品边界

### 3.1 第一阶段必须支持

- SSH 连接配置与测试
- 远端环境探测
- 远端工作目录初始化
- 配置导出并上传到远端
- NapCat 相关文件部署
- 启动 / 停止 / 重启
- 远端状态查询
- 远端日志读取

### 3.2 第一阶段明确不做

- 远端常驻 agent
- 自定义远端开放 API
- 多服务器任务编排平台
- 实时双向推送协议
- 复杂的自动修复系统
- 大规模集群管理

## 4. 架构分层

当前建议采用以下逻辑分层：

### 4.1 本地 Desktop 控制面

负责：

- UI 交互
- SSH 凭据输入
- 本地配置导出
- 远程动作触发
- 状态与日志展示

### 4.2 Remote 基础设施层

负责：

- SSH 连接与 SFTP 上传下载
- 统一执行抽象
- Linux Core 目录初始化
- 远端状态查询与日志读取

当前已落地的代码入口包括：

- `src/core/remote/ssh_client.py`
- `src/core/remote/execution_backend.py`
- `src/core/remote/deployment.py`
- `src/core/remote/status.py`
- `src/core/remote/remote_manager.py`

### 4.3 Shared/Config 层

负责：

- 应用配置项
- Bot 配置导出
- 远端路径布局约定
- 安全默认值

## 5. 远端目录布局约定

第一阶段统一使用用户目录下的专属工作区，默认：

```text
$HOME/NapCatCore/
├── runtime/
│   ├── config/
│   ├── log/
│   ├── tmp/
│   ├── napcat.pid
│   └── status.json
└── packages/
```

设计原则：

- 默认只操作 `$HOME/NapCatCore`
- 不默认写系统目录
- 不要求 root 才能完成第一阶段能力
- 所有运行态元数据尽量收敛到工作区内部

## 6. 安全基线

### 6.1 SSH 凭据策略

- 默认优先使用私钥认证
- 默认拒绝未知主机指纹
- 默认关闭 SSH agent
- 默认关闭自动扫描本地密钥
- 密码不写入配置文件，只允许运行时输入或临时测试使用

### 6.2 权限边界

- 第一阶段默认以普通用户运行
- 远端操作仅针对专属工作目录
- 严禁默认执行高风险全局命令
- 所有删除动作必须限定在工作区路径内

### 6.3 命令策略

- 部署脚本应幂等
- 危险操作要显式确认
- 远端状态优先通过 PID 文件、状态文件、日志文件判断
- 不依赖隐式系统状态作为唯一真相来源

## 7. 分阶段实施计划

### P0：基础设施层

目标：打通 SSH 与远端骨架能力。

包含：

- SSH 客户端封装
- 远程执行抽象
- Linux 目录布局初始化
- 状态 / 日志读取骨架
- 设置页中的连接配置入口

状态：**已完成基础骨架**。

### P1：最小可用版本（MVP）

目标：完成“用户填 SSH 即可管理”的第一版闭环。

包含：

- 连接测试
- 环境探测
- 配置导出并上传
- 远端安装包上传
- 远端部署脚本执行
- 启动 / 停止 / 重启
- 查看日志
- 查看状态

验收标准：

- 用户无需手工 SSH 敲命令即可完成首次接管
- 失败时可以获得明确错误信息
- 本地模式不受远程模式影响

### P2：增强版本

目标：增强远程可维护性和产品完整度。

包含：

- 配置同步回读
- 远端版本检查
- 部署前健康检查
- 部署失败回滚策略
- 多服务器列表管理
- 更完善的日志展示与状态卡片

### P3：高级能力（后续再评估）

仅在 MVP 和增强版稳定后再考虑：

- 远端轻量 helper
- 实时推送
- 后台任务队列
- 批量编排
- 自动修复

## 8. 风险与控制策略

### 8.1 风险：Linux 发行版差异

应对：

- 第一阶段先聚焦主流 Debian / Ubuntu 路径
- 环境探测显式输出系统信息和依赖缺口
- 安装逻辑保持脚本化与可扩展

### 8.2 风险：SSH 凭据管理不当

应对：

- 密码不落盘
- 私钥优先
- 指纹校验默认最严格
- 明确提示用户风险模式（如 `auto_add`）

### 8.3 风险：远端部署失败残留半成品

应对：

- 工作区隔离
- 部署前后写状态文件
- 上传包与运行时目录分离
- 关键步骤记录日志

### 8.4 风险：本地模式被远程改造影响

应对：

- 维持现有本地运行链不变
- 新能力优先挂在 `src/core/remote/`
- 本地与远程通过执行抽象层隔离，而不是直接改写现有运行器

## 10. 架构演进

### 10.1 双模式架构（当前）

项目已演进为支持两种远程管理模式：

**模式一：SSH 直连**
- 适用场景：简单部署、一次性配置
- 特点：无需服务端，直接 SSH/SFTP 操作
- 局限：状态同步依赖轮询，日志实时性受限

**模式二：Agent/Daemon**
- 适用场景：长期监控、实时日志、频繁操作
- 特点：Go Daemon 常驻运行，WebSocket + JSON-RPC 2.0 通信
- 优势：状态实时推送、日志流式传输、自动重连

### 10.2 技术栈

**Daemon (Go)**
- WebSocket: gorilla/websocket
- 协议: JSON-RPC 2.0
- 进程管理: os/exec
- 日志流: 环形缓冲区 + Server-Sent Events

**Desktop (Python)**
- WebSocket: QtWebSockets
- 协议: jsonrpc_protocol.py
- 架构: AgentBackend (ExecutionBackend 子类)

### 10.3 决策结论

**当前路线**：
- 保持 SSH 直连作为基础能力
- 新增 Agent/Daemon 模式作为增强能力
- 用户在远程页可选择连接模式（SSH 或 Agent）
- 两种模式共享 UI 层，通过 ExecutionBackend 抽象隔离

**远景规划**：
- 稳定 Daemon 功能后，Agent 模式可作为默认推荐
- SSH 模式保留作为 fallback 和首次部署手段
